SQL语言概览

SQL语言有以下几部分：

  1. DDL（Data-definition language）数据定义语言
  2. DML（Data-mainpulation Language）数据操作语言
  3. DCL（Data-control Language）数据控制语言

基本类型

char (n) ：长度固定为n的字符串
varchar (n)：长度可变，最大长度为n的字符串，
int：整数
smallint：短整数
numeric (p, q)：总共不多于p位，小数点右边不多于q位的定点数
real：浮点数
float(n)：精度至少为n位的浮点数
date：日期 (年, 月, 日)
time：时间 (小时, 分, 秒)
timetamp：日期+时间
null：每种类型都可以包含一个特殊值，即空值。可以申明属性值不为空，即禁止加入空值

集合运算

SQL作用在关系上的union、intersect和except运算对应于数学集合论中的∪∩−\cup \cap -∪∩−运算；
SQL的集合运算会自动去除重复；
如果想保留重复，必须用union all、intersect all和except all；
假设一个元组在关系R中重复出现了m次，在关系S中重复出现了n次，那么这个元组将会重复出现：

  • R union all S中，重复m+n
  • R intersect all S中，重复min(m, n)
  • R except all S中，重复max(0,m-n)

Order By子句

作用：在select子句得出结果之后，对结果中的元组进行排序
先按属性1进行排序。升序（asc）或者降序（desc），缺省是升序；属性1相等时，再按照属性2的值升序或降序…

总结SQL语句中的运算“次序”

From (笛卡儿积) → [Where (选择)] → [Group By (分组)] → [Having (筛选分组)] → Select (投影, 或统计) → [Order By (结果排序)]

连接语句

连接关系：
在From子句中的两个相邻关系之间，可以是一个逗号 —— 表示做笛卡尔积运算，无条件的连接; 也可以是一个连接运算 —— 表示按指定条件进行连接。
连接运算的类型：

  1. 内连接：
    格式：A [inner] join B on 条件 （A, B是关系）
    内连接的结果不包含失配元组：因不满足连接条件，无法和其它元组相连接的元组。
  2. 左外连接
    格式： A left [outer] join B on 条件
    结果包含左边关系的失配元组
  3. 右外连接
    格式： A right [outer] join B on 条件
    结果包含右边关系的失配元组
  4. 全外连接
    格式： A full [outer] join B on 条件
    结果包含两边关系的失配元组
    注: 通常在From中用逗号分隔关系，然后在Where中列出连接关系的条件。这样的做法相当于内连接运算，即不含失配

视图

视图的概念：一个命名，用查询从基本表中导出的虚拟表；
视图的本质：命名查询，真正存在，存储在数据库里的视图的名字和对应的查询（视图的定义）
视图的表象：“虚拟表”，这样的表逻辑上存在——对用户而言，与基本表无差别；但是物理上不存在——表中的数据平时并没有存储在数据库里，而是用到的时候才算出来；

查询视图

用户可以像基本表一样(写Select语句)查询视图。

  • 系统收到这样的select查询后，必须转化(Why?)成对基本表的查询，方法是展开视图——用定义中的查询取代视图本身
  • 如果展开一个视图后，（它定义中的）查询又还出现了其它视图，那么继续展开，直到没有任何视图为止
    转化的原因是视图物理上是不存在的，无法查询，所以要根据视图的逻辑去物理存储表中查询

修改视图

用户可以像基本表一样(写Insert, Update, Delete语句)修改视图。

  • 系统收到这样的修改后，必须转化成对基本表修改。
  • 这种转化有时能成功——视图可修改，有时失败——视图不可修改。因此我们说对视图的修改是受限的。
  • 能否修改，取决于是否能够通过视图逆映射到基本表中的唯一元组！

不可修改的视图

  1. Select子句选出的属性不包含候选码
    基本表一条记录(有候选码) → 视图一条记录(无候选码)
    因为缺少候选码，所以在逆映射时无法唯一定位原来的记录
  2. select子句中使用distinct关码字
    基本表多条记录 → 视图一条记录
  3. 有group by子句或出现聚集函数
    基本表多条记录 → 视图一条记录
  4. 包含用表达式构造的新属性
    F(基本表字段) → 视图字段（逆映射?）
  5. From子句中列出多个关系
    多个基本表的多条记录 → 视图的一条记录

物化视图

概念：在某些数据库系统中，允许将视图某一时刻的内容（当时的查询结果关系）真正存储起来，这样的“快照” 称为物化视图。

物化视图 (vs 普通视图)
优点：访问快，避免了每次计算(查询结果)的过程。
缺点：只反映基本表当时的查询结果，所以基本表变化后要考虑同步问题。

适用场合：读多于写
经常访问视图，每次重新计算的开销大。但基本表少变化，或者允许“滞后”更新视图的场合。

  3. 非法数据：合法用户合法操作，但是操作错误数据，比如负的年龄；

禁止权限（难点）

对比

grant … to 用户

  • 授予正权限：使用户可以做相应的操作
  • 特殊情况：A，B授予同一个用户同一个权限X，即该用户的权限有两个来源；此时该用户有两个X，当A收回该权限，此用户对应X消失，但他还拥有B给他的X，所以他仍然有权限；

deny … to 用户

  • 禁用（授予负）权限：使用户以后绝对不可以做相应的操作
  • 比如上面的例子，C禁止该用户的X权限，即给了用户一个-X，那么用户无法再做此权限的操作，即使之前A，B授予了“正权限X”；
  • 所以负权限大于正权限，只要有一个负权限，无论得到多少正权限都无法获得权限

revoke … from 用户

  • 收回，相当于取消此前授予的（正、负）权限；注意：取消禁止权限，也是通过revoke
  • 思考：如果B收回授予用户的正权限X，有什么效果？接下来C取消他授予的负权限（禁止权限）呢？
    现在用户有两个X和一个-X，B收回后，用户有一个X和一个-X，仍然无法使用权限；
    C撤销负权限后，用户有一个X，无-X，则拥有权限；

小结：一个用户无法使用权限的情况：1. 无权限；2. 被授予负权限；

角色机制

“角色”出现的必要性：
如果有一组用户，他们的权限相同或非常相近，那么设置一个角色代表这组用户，同意给角色授权，可以大大减少工作量；

角色的概念：

  1. 角色就是一个用户集合
  2. 角色也是一个权限集合

例子：
辅导员小王、小李、…想访问数据库里面的学生表；但只能允许他们查询和更新某些字段，比如年龄、宿舍。

角色即可以出现在to前面，又可以出现在to后面，所以它即是权限集合，又是用户集合

思考：
如果所有辅导员权限不完全一致，比如某一个辅导员比其他辅导员权限多一点或少一点，那还可以应用角色吗？怎么处理？
多一点：按照用户赋给额外的权限
少一点：deny掉该用户的权限

完整性约束

概念：数据的正确性、有效性和相容性
正确的数据就是具有完整性的

完整性规则：

  • 为了保证完整性，数据所必须满足的约束条件；（又称完整性约束、完整性约束条件）
  • 可以分为两类：
    1. 静态条件：数据取值的条件；比如年龄大于等于0
    2. 动态约束：数据变化/修改数据的条件（新旧值的比较）
      例如：工资.新值 < 工资.旧值*1.2
      我们大多数讲完整性，都是在讲完整性规则

完整性控制（实现完整性）：

  • 在数据库系统中，保证数据完整性，这一工作由DBMS完整性子系统完成（可能考选择题填空题）
  • 围绕完整性规则进行，包括两方面：
    1. 定义完整性规则（SQL，4.4）
    2. 检查完整性规则，并做处理
      比如，在用户发出操作（insert、update等）后，检查修改数据后是否还满足完整性规则，如果不能则进行处理：报警、拒绝、调整

关系模型的三种完整性概念

关系模型的三要素：

  • 关系
  • 关系操作
  • 关系完整性 ——> 指的是完整性规则
    1. 实体完整性规则
    2. 参照完整性规则
    3. 用户定义完整性规则

实体完整性规则

规则：关系中，元组的主码必须唯一不重复，且不能取空值；若主码由多个属性构成，每一个属性都不能取空值；
意义：现实世界实体可以相互区分，实体映射到元组后，通过主码来相互区分；

参照完整性规则

规则：外部码要么取空值，要么取一个存在的对应的主码值；
意义：外部码是用于反映联系；取空值，代表没有联系；取一个对应的主码值，表示外部码所在元组/实体和这个主码值所在元组/实体联系；

思考：创建表时，外部码中的属性应该设为什么类型？
外部码的属性类型要严格等于主码字段类型

用户定义完整性

规则：用户根据具体的应用环境定义
例如：

  1. 分数的取值范围：0-100
  2. 性别只能：男、女、空
  3. （上海）A股股东账号是“A+9位数字”
    意义：
  4. 反映现实世界的真实属性取值
  5. 反映了程序编制的需求

什么会破坏完整性规则

  1. 显然查询操作不会破坏完整性，只有改变数据的修改操作才可能

各种约束

主码约束
  1. 规则：主码值不允许为空，也不允许重复
  2. 定义：创建/修改表的语句中，用PRIMARY KEY声明主码的同时定义一个主码约束；
外部码约束
  1. 规则：外部码要求取值要么空值要么取对应的主码值
  2. 定义：创建/修改表的语句中，用FOREIGN KEY声明外部码的同时，即定义一个外部码约束
  1. 参照动作：说明当（被参照关系的）某个主码值被删除/更新时，如何处理自己表中对应的外部码值
    RESTRCT方式（for server: no action）：仅当没有任何对应外部码值时才能删除/更新这个主码值，否则系统拒绝执行此操作
    CASCADE方式：连带所有对应的外部码值一起删除/更新
    SET NULL方式：将所有外部码值设为空
    SET DEFALUT方式：将所有对应外部码值设置为默认值

例子：

非空约束

规则：属性值不允许取空值
定义：创建/修改表的语句中，声明某个属性NOT NULL

默认值约束

规则：插入记录时，没有指定数据的属性值取默认值
定义：创建/修改表的语句中，声明某个属性时，用DEFAULT关键字引出其默认值

唯一约束

规则：制定属性或属性组，不能取重复值
定义：创建/修改表的语句中，用Unique关键字声明

检查约束

规则：每个元组内部一个或多个属性值，必须满足用户指定的条件
定义：创建/修改表的语句中，check关键字后说明条件

域约束

规则：域（数据类型）中的值要满足用户指定的条件
目的：间接地约束属性。因为如果把某个属性声明为这个域（数据类型），则该属性的取值也连带要满足这一条件。
定义：在创建域的语句中，在constraint关键字后面说明约束的名称和条件。

断言约束

断言就是一个谓词（条件），实施在较大的范围，例如整个数据库所有数据

  • 主码约束、检查约束等作用范围较小，一般是一个元组或者属性内部，可以看做是“小断言”
  • 如果作用范围比较大，例如要求几个关系之间必须满足什么约束条件，就要断言约束
    创建断言：
  • 创建断言后，系统要对每个可能违反该断言条件的修改操作进行检查，这种检查会带来巨大的系统负载，因此应该谨慎使用断言
  • 所以实际上很少数据库软件支持断言，而倾向于用其他的等价方法，比如触发器
触发器

概念：自动执行的过程

  • 传统过程是用户调用才手动执行，否则不执行
  • 在系统检测到某些事情发生时才会去主动执行相应（预先跟这些时间绑定的）触发器
    构成：
  • Event ： 发生什么是？
  • Condition ： 检查什么条件？
  • Action：再执行什么动作（条件成立）？
    作用：
  1. 保证数据完整性，例如在增删改记录时，验证一些复杂的条件是否满足
  2. 审计（例如把一个用户对数据库的操作和时间都记录下来）
  3. 满足特定条件自动执行某项任务（例如，商品库存不足时，自动生成一张订单）

例子：当职工工资增幅超过10%，放弃修改（改回原值）

小结

安全性和完整性的对比

  • 安全性控制是为了防止“非法用户”和“非法操作”（用户合法但执行的操作超出权限范围）
  • 完整性控制是为了防止“非法数据”，也就是不正确的数据。任何情况下，即使是合法用户以及合法操作，也会产生非法数据。
